```python
def actor_arc_length_pre_window(
    actions: pd.DataFrame,
    frames: pd.DataFrame,
    *,

    Per-action sum of consecutive segment distances over frames in
    (action_time - pre_seconds, action_time], filtered to the actor's
    player_id within the same period:

        Î£_{k=1..N-1} âˆš((x_{k+1} - x_k)Â² + (y_{k+1} - y_k)Â²)

    Bauer & Anzer 2021 counterpressing detection (5s), pass pre_seconds=5.0.

    sprint-intensity filtering. See NOTICE.
```

### 3.2 NaN semantics â€” the "bridge" rule

NaN-position frames are dropped entirely BEFORE computing segments or
selecting first/last. This is the **"bridge" rule**: NaN gaps are bridged
by treating consecutive valid frames as adjacent for segment-summing.
Rationale: tracking-NaN typically means "tracker briefly lost the player",
not "player teleported to NaN"; assuming the player was where they ended
up is more informative than dropping the segment.

| Window state | arc-length | displacement |
|---|---|---|
| Action couldn't link (period gap, no frames, etc.) | NaN | NaN |
| 0 valid-position frames in window for actor's player_id | NaN | NaN |
| Exactly 1 valid-position frame in window | NaN | NaN |
| â‰¥2 valid-position frames | sum of segments between consecutive valid frames | âˆš((x_last_valid - x_first_valid)Â² + (y_last_valid - y_first_valid)Â²) |

Concrete pin examples (4 valid frames at 0.0/0.1/0.3/0.4 seconds before action):

| Frames in window (positions @ time offsets) | arc-length expected | displacement expected |
|---|---|---|
| (0,0)@-0.4, (1,0)@-0.3, (2,0)@-0.2, (3,0)@-0.1, (4,0)@0 | 4.0 | 4.0 |
| (0,0)@-0.4, (1,0)@-0.3, NaN@-0.2, (3,0)@-0.1, (4,0)@0 | 4.0 (segments 0â†’1, 1â†’3 [bridge], 3â†’4 = 1+2+1) | 4.0 |
| (0,0)@-0.4, NaN@-0.3, NaN@-0.2, (3,0)@-0.1 | 3.0 (single bridged segment 0â†’3) | 3.0 |
| (0,0)@-0.4 only (rest fall outside window) | NaN | NaN |
| 0 frames in window | NaN | NaN |

Public function never raises on NaN-position rows (per existing 19-column
tracking schema invariant 5).

Rationale for one shared kernel emitting both quantities: arc-length and displacement share the same expensive operation (`slice_around_event` + filter to actor's player_id + sort + drop-NaN-positions); computing both in one pass avoids duplicate work when consumers want both. The two single-quantity public defs slice columns from the shared kernel output. Mirrors `_pre_shot_gk_position` (PR-S21) and `_pre_shot_gk_angle` (PR-S24) emit-multiple-columns pattern.

### 4.4 Kernel internals (schema-agnostic, in `_kernels.py`)

```python
def _pressure_andrienko(
    anchor_x: pd.Series,
    anchor_y: pd.Series,
    ctx: ActionFrameContext,
    *,
    params: AndrienkoParams,
    goal_x: float = _GOAL_X,            # = 105.0 from existing _kernels._GOAL_X
    goal_y: float = _GOAL_Y_CENTER,     # = 34.0  from existing _kernels._GOAL_Y_CENTER
) -> pd.Series:
    """Andrienko sum-of-pressers oval model.

    Per defender:
      vec_to_threat = (goal - anchor) / ||goal - anchor||
      vec_presser   = (defender - anchor) / ||defender - anchor||
      cos_theta     = dot(vec_to_threat, vec_presser)
      z             = (1 + cos_theta) / 2
      L             = d_back + (d_front - d_back) * (z**3 + 0.3*z) / 1.3
      d             = ||defender - anchor||
      pr_i          = max(0, 1 - d/L) ** q  * 100.0   if d < L else 0.0

    Aggregation: sum across all opp-team defenders (excluding ball rows).
    """


def _pressure_link(
    anchor_x: pd.Series,
    anchor_y: pd.Series,
    ctx: ActionFrameContext,
    *,
    params: LinkParams,
    goal_x: float = _GOAL_X,           # SPADL-LTR canonical (105.0)
    goal_y: float = _GOAL_Y_CENTER,    # SPADL-LTR canonical (34.0)
) -> pd.Series:
    """Link 2016 piecewise-zone saturating-aggregation model.

    Per defender:
      d            = ||defender - anchor||
      cos_theta    = ((goal - anchor) Â· (defender - anchor)) / (||..|| * ||..||)
      alpha_deg    = degrees(acos(clip(cos_theta, -1, 1)))   # in [0, 180]
      r_zo         = r_hoz if alpha_deg < angle_hoz_lz_deg
                     else r_lz if alpha_deg < angle_lz_hz_deg
                     else r_hz
      pr_i         = max(0, 1 - d/r_zo)   if d < r_zo else 0.0
      # No HPZ special case â€” Eq (3) is the formal spec; prose's "HPZ constant
      # high pressure" is a qualitative observation honored implicitly by the
      # monotone formula. See LinkParams docstring NOTE 1.

    Aggregation: PR(x) = 1 - exp(-k3 * x) where x = sum(pr_i).
    """


def _pressure_bekkers(
    anchor_x: pd.Series,
    anchor_y: pd.Series,
    ctx: ActionFrameContext,
    *,
    params: BekkersParams,
    ball_xy_v_per_action: pd.DataFrame | None,
) -> pd.Series:
    """Bekkers Pressing Intensity probabilistic model.

    For each action, per defender i with position p_i, velocity v_i:
      d2     = anchor + v_actor   # naive 1-second-extrapolated target position
      angle  = arccos(v_i Â· (d2 - p_i) / (||v_i|| * ||d2 - p_i|| + eps))
      r_react = p_i + v_i * tau_r
      tti    = ||v_i|| * angle / pi + tau_r + ||d2 - r_react|| / v_max
      p_to_player = 1 / (1 + exp(-pi/sqrt(3)/sigma * (T - tti)))
      if speed_i < speed_threshold:
          p_to_player = 0

    If use_ball_carrier_max and ball position+velocity available:
      p_to_ball = same formula with anchor=ball_pos, v_actor=ball_vel
      p_i_final = max(p_to_player, p_to_ball)
    Else:
      p_i_final = p_to_player

    Aggregation: P_j = 1 - prod(1 - p_i_final).

    Re-implementation with attribution; canonical reference:
    UnravelSports/unravelsports/unravel/soccer/models/utils.py (BSD-3-Clause).
    """
```

The Bekkers kernel takes a separate `ball_xy_v_per_action` argument (one row per action with ball x/y/vx/vy at the linked frame, or NaN if no ball row). Built once at the call site by the public function (or by `add_pressure_on_actor`) so the kernel stays pure.

```
### 8. Multi-flavor xfn column naming convention

When a single feature concept admits multiple methodologies (multi-flavor
xfns â€” first concrete instance: PR-S25 `pressure_on_actor`):

- Each flavor MUST emit a flavor-suffixed column name `<feature>__<method>`
  (double-underscore separator) so consumers registering parallel
  `functools.partial(fn, method="X")` xfns in `VAEP.xfns` do not silently
  overwrite each other inside `VAEP.compute_features`.
- The default xfn list (`<feature>_default_xfns`) ships exactly ONE flavor
  (the default method) to keep the VAEP feature space stable across silly-kicks
  versions. Consumers wanting additional flavors register additional
  `functools.partial` xfns explicitly.
- Per-method parameters are passed via a flavor-specific frozen dataclass
  (e.g., `AndrienkoParams`, `LinkParams`, `BekkersParams`) on the `params=`
  kwarg, not as a flat keyword bag â€” this keeps each flavor's parameter
  surface discoverable and statically typed (`pyright`-friendly), and
  allows `__post_init__` validation per flavor.
- If `params=` is supplied with a type not matching the chosen `method`, the
  public function raises TypeError loudly (no silent default fallback).

This pattern applies only to **VAEP-consumed xfns** that emit per-action
scalar features. Preprocessing utilities that produce canonical per-row
columns (e.g., `derive_velocities` emits `vx`, `vy`, `speed`) keep their
canonical-single-column names regardless of method; QA/inspection helpers
case-by-case (per `feedback_multi_flavor_xfn_column_names`).

Rationale recap: the suffix convention surfaces the methodology choice in
the column name (downstream debugging is easier â€” column-name self-documents
which formula produced each value), and the default-xfn-list-ships-one-flavor
rule prevents "oh, did adding Link to the default break my model artefact?"
regression after consumer updates.
```
